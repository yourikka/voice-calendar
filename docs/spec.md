# 语音版日历工具 Spec

## 文档信息

- 项目名称：语音版日历工具
- 文档类型：产品与技术规格说明
- 当前阶段：需求定义与 MVP 设计
- 目标读者：产品负责人、前后端工程师、语音/NLU 工程师、测试工程师
- 核心目标：通过自然、准确、可纠错的语音交互，降低用户创建、查看、删除和管理日历事件与提醒的操作成本。

## 1. 背景与目标

### 1.1 问题定义

传统日历工具依赖手动输入，用户在移动、开车、做饭、办公切换等场景下很难快速完成日程管理。真实使用中，用户往往不是想“打开日历并填写表单”，而是想直接表达：

- “明天下午三点提醒我去医院复诊”
- “下周二上午十点到十一点和产品开会”
- “帮我看看今天晚上有没有空”
- “把明天的健身取消”
- “提前半小时提醒我周五的面试”

现有日历工具常见痛点：

- 添加事件链路长，需要手动选择日期、时间、标题、提醒
- 自然语言时间表达复杂，例如“下周一”“月底前”“每隔两周”“工作日下午”
- 删除或修改事件时容易误删，缺少确认和撤销
- 查看日程时屏幕信息密集，不适合碎片化语音查询
- 多日历、多时区、重复事件、冲突事件处理复杂
- 语音识别错误后，用户很难快速修正

### 1.2 产品目标

构建一个以语音为核心入口的日历管理工具，让用户能用自然语言完成日程管理。

核心目标：

- 支持语音添加、删除、修改、查看事件
- 支持语音创建、更新、取消提醒
- 支持语音获取当日资讯，并可与当天日程组合成每日简报
- 支持实时从网络抓取当日热点，并在日历界面展示“今日热点”专栏
- 准确理解中文自然语言日期、时间、重复规则和提醒规则
- 对不完整或有歧义的指令进行澄清，而不是盲目执行
- 对高风险操作进行确认，并提供撤销能力
- 与系统日历或第三方日历同步，避免形成信息孤岛
- 在移动端、桌面端和车载/耳机等免手持场景中保持可用

### 1.3 非目标

以下能力不作为 MVP 必须交付项：

- 企业级会议室预订系统
- 完整团队排班和复杂资源调度
- 邮件自动解析并创建会议
- 深度 CRM 或项目管理集成
- 跨组织会议自动找时间
- 离线大模型本地全量推理
- 多人共享家庭日历的复杂权限模型
- 完整新闻客户端、舆情监控平台或自媒体内容生产系统

## 2. 用户研究与真实需求

### 2.1 目标用户

#### 个人效率用户

特点：

- 日程密集，频繁创建临时提醒
- 关注快速录入、快速查看、避免遗漏
- 常用场景是通勤、办公切换、睡前安排第二天

典型需求：

- 快速添加提醒
- 查看今天、明天、本周安排
- 找空档
- 修改时间
- 删除临时事项

#### 职场会议用户

特点：

- 会议多，时间变动频繁
- 需要区分工作日历、个人日历
- 对冲突检测、提前提醒、会议链接敏感

典型需求：

- 创建会议类事件
- 识别参会人、地点、线上会议链接
- 检查时间冲突
- 修改或取消会议
- 按项目、对象、日期查看日程

#### 生活提醒用户

特点：

- 主要使用提醒，而不是完整会议日程
- 表达更口语化
- 时间可能模糊，例如“晚上”“睡前”“下班后”

典型需求：

- 创建单次提醒
- 创建重复提醒
- 提前提醒
- 按标签查看，例如家庭、健康、缴费

#### 免手持场景用户

特点：

- 正在开车、做饭、运动或走路
- 不方便阅读长列表
- 更依赖语音播报和简短确认

典型需求：

- 低交互成本
- 高容错
- 语音确认
- 支持取消刚才操作

### 2.2 核心用户任务

按用户真实意图排序，MVP 优先覆盖：

1. 添加事件或提醒
2. 查看某段时间日程
3. 删除事件或提醒
4. 修改事件时间、标题、地点或提醒
5. 查询是否有空
6. 创建重复事件
7. 处理时间冲突
8. 撤销最近一次操作

### 2.3 用户成功标准

用户认为工具好用的关键不是“能识别所有语音”，而是：

- 说一句话就能完成大多数日程操作
- 不确定时会问清楚
- 执行前能复述关键字段
- 错了能快速改
- 删除和覆盖不会误操作
- 查看结果短、准、可继续追问
- 早晨或通勤时能一句话获取当天日程和重要资讯

## 3. 产品范围

### 3.1 MVP 功能范围

MVP 必须支持：

- 语音输入转文本
- 文本意图识别
- 日期时间解析
- 添加单次事件
- 添加单次提醒
- 添加简单重复事件
- 查看日程
- 删除事件
- 修改事件
- 设置提前提醒
- 冲突检测
- 澄清式对话
- 操作确认
- 最近操作撤销
- 获取当日资讯
- 生成日程与资讯组合的每日简报
- 日历界面展示今日热点专栏
- 后端实时抓取或聚合当天热点数据，并做短期缓存
- 日历数据本地存储
- 至少一种系统日历同步方案

### 3.2 后续增强范围

后续版本可增加：

- 多平台系统日历双向同步
- Google Calendar、Outlook Calendar、Apple Calendar 集成
- 智能找时间
- 根据上下文自动补全地点、参会人
- 会议纪要关联日程
- 多语言输入
- 离线语音识别
- 个性化时间偏好学习
- 家庭共享日历
- 自然语言批量操作
- 资讯来源管理和个性化频道订阅
- 基于日程地点、行业和兴趣的资讯推荐

## 4. 核心使用场景

### 4.1 添加事件

用户说：

```text
下周三下午两点到三点和李雷开项目评审会，提前十分钟提醒我
```

系统应解析：

- 意图：创建事件
- 标题：项目评审会
- 时间：下周三 14:00-15:00
- 参与人：李雷
- 提醒：提前 10 分钟

系统回复：

```text
已准备创建：下周三下午 2 点到 3 点，项目评审会，参与人李雷，提前 10 分钟提醒。需要保存吗？
```

用户确认后保存。

### 4.2 添加提醒

用户说：

```text
明早八点提醒我带身份证
```

系统应创建提醒类事件：

- 标题：带身份证
- 时间：明天 08:00
- 类型：reminder
- 默认无持续时间

系统回复：

```text
已设置明天早上 8 点提醒你带身份证。
```

### 4.3 查看日程

用户说：

```text
我今天下午有什么安排？
```

系统查询今天 12:00-18:00 的事件，按时间升序播报：

```text
今天下午有 2 个安排：2 点到 3 点项目评审会，4 点半牙医复诊。
```

如果没有事件：

```text
今天下午没有已安排的日程。
```

### 4.4 删除事件

用户说：

```text
取消明天下午的健身
```

系统查询匹配事件：

- 如果唯一匹配：要求确认
- 如果多个匹配：列出候选并要求选择
- 如果无匹配：提示未找到，并允许换说法

确认回复：

```text
找到明天下午 6 点的健身。确认取消吗？
```

执行后：

```text
已取消。你可以说“撤销刚才操作”恢复。
```

### 4.5 修改事件

用户说：

```text
把周五上午的面试改到下午三点
```

系统应：

1. 根据“周五上午的面试”查找事件
2. 如果唯一匹配，更新开始时间
3. 保持原持续时间不变
4. 检查新时间是否冲突
5. 要求用户确认

如果原事件是 10:00-11:00，则更新为 15:00-16:00。

### 4.6 查询空闲时间

用户说：

```text
明天下午三点到五点我有空吗？
```

系统应查询该时间段内事件。

有冲突时：

```text
明天下午 3 点到 5 点不完全空闲。你 3 点半到 4 点有产品同步会。
```

无冲突时：

```text
明天下午 3 点到 5 点是空的。
```

### 4.7 重复事件

用户说：

```text
每周一三五晚上八点提醒我跑步
```

系统应解析：

- 类型：重复提醒
- 规则：每周一、周三、周五
- 时间：20:00
- 标题：跑步

系统需要确认结束条件：

```text
要设置为每周一、三、五晚上 8 点提醒跑步。需要一直重复，还是设置结束日期？
```

MVP 可以默认支持“无限重复”和“直到某日期”。

### 4.8 获取当日资讯

用户说：

```text
今天有什么重要新闻？
```

系统应：

1. 基于用户时区确定“当日”
2. 拉取配置来源中的最新资讯
3. 按重要性、时效性和用户偏好排序
4. 生成适合语音播报的短摘要
5. 返回来源名称和可打开链接

系统回复：

```text
今天有 5 条重要资讯。第一，国内财经方面，央行发布新的货币政策执行报告。第二，科技方面，多家 AI 公司发布模型更新。要我继续详细说哪一条？
```

如果用户指定领域：

```text
今天科技圈有什么新闻？
```

系统只返回科技相关资讯。

### 4.9 每日简报

用户说：

```text
播报一下我的今日简报
```

系统应组合：

- 今日日程摘要
- 待提醒事项
- 当日重要资讯
- 可选天气或通勤提示，MVP 可不实现

系统回复：

```text
今天你有 3 个日程，最早是上午 10 点的周会。还有 2 条提醒。今日重要资讯有 5 条，我先播报前三条。
```

每日简报不应过长。默认先播报摘要，用户追问后再展开。

### 4.10 日历今日热点专栏

用户打开日历首页或当日视图时，系统应在日历中展示一个独立的“今日热点”专栏。

专栏展示内容：

- 今日热点标题
- 简短摘要
- 来源名称
- 发布时间
- 热度或排序信号
- 原文链接
- 刷新时间

交互要求：

- 专栏不替代语音获取热点能力，用户仍然可以说“今天有什么热点”来获得语音回复
- 专栏不占用日程事件格子，避免把资讯误认为用户日程
- 点击热点后打开详情或来源链接
- 支持手动刷新
- 刷新失败时展示上次成功抓取的缓存，并提示缓存时间

界面示例：

```text
今日热点
1. AI 多模态应用持续升温
   来源：示例科技，09:20 更新
2. 国内财经政策发布新动态
   来源：示例财经，08:40 更新
3. 本地交通发布晚高峰提示
   来源：示例本地，08:10 更新
```

语音示例：

```text
用户：今天有什么热点？
系统：今天有 5 条热点。我先说前三条：第一，AI 多模态应用持续升温。第二，国内财经政策发布新动态。第三，本地交通发布晚高峰提示。
```

## 5. 语音交互设计

### 5.1 交互原则

- 默认短确认，避免冗长播报
- 高风险操作必须确认
- 缺少关键字段必须追问
- 多候选必须让用户选择
- 支持用户打断
- 支持“算了”“取消”“重新来”
- 支持“改成明天”“不是三点，是四点”等增量修正
- 任何写操作完成后支持撤销

### 5.2 对话状态

系统至少需要维护以下会话状态：

- `idle`：等待用户输入
- `collecting_slots`：补全字段
- `awaiting_confirmation`：等待确认
- `selecting_candidate`：多候选选择
- `executing`：执行日历操作
- `completed`：操作完成
- `cancelled`：用户取消
- `error_recovery`：错误恢复

### 5.3 澄清策略

缺少日期：

```text
用户：提醒我交房租
系统：什么时候提醒你？
```

缺少时间：

```text
用户：明天提醒我交房租
系统：明天几点提醒？
```

时间模糊：

```text
用户：晚上提醒我吃药
系统：晚上几点？你也可以说“默认晚上 8 点”。
```

删除多候选：

```text
用户：取消明天的会
系统：明天有 3 个会议：上午 10 点周会，下午 2 点评审会，下午 5 点复盘会。你要取消哪一个？
```

### 5.4 确认策略

必须确认的操作：

- 删除事件
- 修改事件时间
- 修改重复事件
- 批量删除
- 覆盖已有提醒
- 创建存在冲突的事件
- 向第三方日历写入数据

可以免确认的操作：

- 查看日程
- 查看空闲
- 创建低风险单次提醒
- 创建字段完整且无冲突的个人事件，前提是用户开启“快速创建模式”

### 5.5 语音播报策略

播报长度控制：

- 0 个事件：一句话
- 1-3 个事件：全部播报
- 4 个以上事件：先摘要，再询问是否继续

示例：

```text
明天有 6 个安排，上午 2 个，下午 3 个，晚上 1 个。要我从早到晚念一遍吗？
```

## 6. 意图与槽位规格

### 6.1 意图列表

| 意图 | 标识 | 示例 |
| --- | --- | --- |
| 创建事件 | `create_event` | “明天下午三点开会” |
| 创建提醒 | `create_reminder` | “晚上八点提醒我吃药” |
| 查看日程 | `list_events` | “今天有什么安排” |
| 查看空闲 | `check_availability` | “明天三点有空吗” |
| 删除事件 | `delete_event` | “取消周五的面试” |
| 修改事件 | `update_event` | “把会议改到四点” |
| 设置提醒 | `set_reminder` | “这个会提前半小时提醒” |
| 取消提醒 | `remove_reminder` | “取消明天会议的提醒” |
| 重复规则 | `set_recurrence` | “每周一提醒我提交周报” |
| 获取当日资讯 | `get_today_news` | “今天有什么重要新闻” |
| 获取每日简报 | `get_daily_briefing` | “播报一下今日简报” |
| 选择候选 | `select_candidate` | “第二个” |
| 确认 | `confirm` | “确认” |
| 否定 | `reject` | “不是” |
| 取消当前流程 | `cancel_flow` | “算了” |
| 撤销操作 | `undo_last_action` | “撤销刚才操作” |

### 6.2 核心槽位

| 槽位 | 类型 | 必填场景 | 说明 |
| --- | --- | --- | --- |
| `title` | string | 创建/修改/删除 | 事件标题或提醒内容 |
| `date_expression` | string | 创建/查询/删除 | 原始日期表达 |
| `time_expression` | string | 创建/查询/修改 | 原始时间表达 |
| `start_at` | datetime | 创建/查询/修改 | 解析后的开始时间 |
| `end_at` | datetime | 事件创建 | 解析后的结束时间 |
| `duration_minutes` | integer | 可选 | 未给结束时间时推断 |
| `location` | string | 可选 | 地点 |
| `participants` | string[] | 可选 | 参与人 |
| `calendar_id` | string | 可选 | 目标日历 |
| `reminders` | object[] | 可选 | 提醒规则 |
| `recurrence_rule` | object | 重复事件 | 重复规则 |
| `news_category` | string | 资讯查询 | 资讯类别，例如财经、科技、体育 |
| `news_region` | string | 资讯查询 | 地域范围，例如本地、国内、国际 |
| `news_limit` | integer | 资讯查询 | 返回条数 |
| `briefing_sections` | string[] | 每日简报 | 简报模块，例如日程、提醒、资讯 |
| `timezone` | string | 默认必填 | 用户当前时区 |
| `confidence` | number | 内部字段 | 识别置信度 |

### 6.3 时间表达解析

必须支持的日期表达：

- 今天、明天、后天
- 本周、下周、下下周
- 周一到周日
- 这个周末、下个周末
- 具体日期，例如“6 月 3 号”
- 相对日期，例如“三天后”
- 月初、月底、下个月初
- 工作日、周末

必须支持的时间表达：

- 上午、下午、晚上、凌晨
- 具体时间，例如“三点半”“15 点 20”
- 时间范围，例如“下午两点到三点”
- 相对时间，例如“半小时后”
- 模糊时间，例如“下班后”“睡前”，MVP 需要澄清或使用用户默认值

默认时间段建议：

| 表达 | 默认时间段 |
| --- | --- |
| 早上 | 07:00-09:00 |
| 上午 | 09:00-12:00 |
| 中午 | 12:00-13:30 |
| 下午 | 13:00-18:00 |
| 晚上 | 18:00-22:00 |
| 睡前 | 用户配置，默认 22:30 |
| 下班后 | 用户配置，默认 18:30 |

### 6.4 重复规则解析

MVP 支持：

- 每天
- 每周某几天
- 每月某日
- 工作日
- 周末
- 直到某日期结束
- 永不结束

内部建议使用 RFC 5545 RRULE 表达：

```json
{
  "freq": "WEEKLY",
  "byday": ["MO", "WE", "FR"],
  "interval": 1,
  "until": null
}
```

## 7. 数据模型

### 7.1 Event

```json
{
  "id": "evt_01HZY...",
  "calendar_id": "primary",
  "type": "event",
  "title": "项目评审会",
  "description": "",
  "start_at": "2026-06-03T14:00:00+08:00",
```

## 8. 系统架构

### 8.1 逻辑架构

```text
客户端
  -> 语音采集
  -> ASR 语音转文本
  -> NLU 意图与槽位解析
  -> 对话管理器
  -> 日历编排服务 / 资讯服务 / 每日简报服务
  -> 日历存储 / 第三方日历适配器 / 资讯来源适配器
  -> TTS 语音播报
```

### 8.2 模块职责

#### VoiceCapture

职责：

- 采集用户语音
- 支持按住说话和唤醒后连续对话
- 处理静音检测
- 上传音频或调用本地 ASR

#### ASRService

职责：

- 将语音转换为文本
- 返回候选文本和置信度
- 保留时间戳用于调试

输出示例：

```json
{
  "transcript": "明早八点提醒我带身份证",
  "alternatives": [],
  "confidence": 0.93
}
```

#### NLUService

职责：

- 识别用户意图
- 抽取槽位
- 解析日期时间
- 标准化标题、地点、参与人
- 给出置信度和缺失字段

输出示例：

```json
{
  "intent": "create_reminder",
  "confidence": 0.91,
  "slots": {
    "title": "带身份证",
    "start_at": "2026-05-30T08:00:00+08:00",
    "timezone": "Asia/Shanghai"
  },
  "missing_slots": []
}
```

#### DialogManager

职责：

- 管理多轮对话状态
- 决定是否需要澄清
- 决定是否需要确认
- 支持用户修正和取消
- 将完成的用户意图转换为可执行命令

#### CalendarOrchestrator

职责：

- 执行创建、修改、删除、查询
- 检查冲突
- 处理重复事件
- 写入操作日志
- 提供撤销

#### CalendarStore

职责：

- 本地事件存储
- 查询索引
- 重复事件展开
- 软删除

#### CalendarAdapter

职责：

- 对接系统日历或第三方日历
- 处理同步状态
- 处理远端 ID 映射
- 处理冲突和失败重试

#### NewsService

职责：

- 获取当日资讯和实时热点
- 按类别、地区和语言过滤资讯
- 对多来源资讯去重
- 生成适合语音播报的摘要
- 返回来源名称、发布时间和原文链接

#### NewsProviderAdapter

职责：

- 对接 RSS、新闻 API、热榜接口或合规网页来源
- 统一不同来源的数据结构
- 处理请求限流、失败重试和缓存
- 标记来源可信度和内容时效性

#### HotTopicCrawler

职责：

- 按配置源实时抓取当日热点
- 遵守来源站点 robots、频率限制和使用条款
- 抽取标题、摘要、发布时间、来源、链接和热度信号
- 对相同热点做跨来源合并
- 将结果写入短期缓存，供语音和日历专栏复用

#### CalendarHotTopicPanelService

职责：

- 为日历首页或当日视图提供“今日热点”专栏数据
- 控制专栏展示条数和排序
- 读取热点缓存，必要时触发后台刷新
- 在抓取失败时返回上次成功缓存
- 确保热点专栏不写入用户日程事件

#### DailyBriefingService

职责：

- 聚合当日日程、提醒和资讯
- 控制播报长度
- 支持用户追问“详细说第二条”
- 根据用户偏好调整简报顺序

### 8.3 推荐技术栈

MVP 可采用：

- 客户端：Web/PWA 或移动端原生壳
- 后端：FastAPI / Node.js 均可
- 数据库：SQLite 起步，后续迁移 PostgreSQL
- ASR：系统语音 API 或云语音识别
- NLU：规则解析 + LLM 辅助解析混合方案
- 时间解析：自研规则层 + chrono 类库能力
- TTS：系统 TTS
- 同步：先支持本地 ICS 导出或系统日历写入，再扩展 Google/Outlook
- 资讯：RSS/新闻 API/合规热榜抓取 + 本地短期缓存 + 摘要生成
- 热点缓存：SQLite/Redis 均可，MVP 可使用 SQLite 表加 TTL 字段

## 10. 关键业务规则

### 10.1 默认持续时间

当用户只说开始时间但没有结束时间：

- 会议类事件默认 60 分钟
- 面试默认 60 分钟
- 健身默认 60 分钟
- 吃饭默认 90 分钟
- 提醒类事件无持续时间
- 用户可在设置中覆盖默认值

### 10.2 冲突检测

创建或修改事件时必须检测同一日历下的时间重叠。

冲突时系统回复：

```text
这个时间和下午 3 点半到 4 点的产品同步会冲突。仍然保存吗？
```

用户确认后允许保存，但事件应记录 `has_conflict=true` 或在响应中返回冲突信息。

### 10.3 删除匹配规则

删除事件时按以下顺序匹配：

1. 时间范围
2. 标题关键词
3. 类型
4. 地点或参与人
5. 最近创建或最近更新

置信度不足或候选超过 1 个时必须让用户选择。

### 10.4 修改重复事件

修改重复事件时必须询问作用范围：

```text
这是一个重复日程。你要只改这一次，还是改之后所有重复项？
```

MVP 支持：

- 只修改本次
- 修改所有未来事件

### 10.5 撤销规则

必须支持撤销：

- 最近一次创建
- 最近一次删除
- 最近一次修改

默认撤销窗口：

- 本地操作：10 分钟
- 第三方同步操作：以远端能力为准，至少本地回滚并提示远端状态

### 10.6 当日资讯规则

当日资讯必须以用户时区为准，不直接使用服务器日期。

当用户通过语音询问热点时，系统仍然走资讯语音播报链路。日历专栏只是新增展示入口，不能取消或替代已设置的语音回复热点能力。

资讯结果排序建议：

1. 发布时间越新优先级越高
2. 来源可信度越高优先级越高
3. 与用户指定类别或地区匹配优先级越高
4. 同一事件多来源报道需要合并或去重
5. 热榜排名、搜索热度或传播热度越高优先级越高

语音播报规则：

- 默认最多播报 3 条摘要
- 返回列表最多 10 条
- 每条摘要控制在 1-2 句
- 必须保留来源名称和链接
- 用户追问“详细说第二条”时再展开

实时抓取规则：

- 默认缓存 TTL 为 5-10 分钟
- 用户手动刷新可以绕过缓存，但必须受频率限制
- 抓取失败时返回上次成功缓存并标记 `stale=true`
- 严禁在抓取失败时编造热点
- 只接入白名单来源，避免无边界爬取
- 必须遵守目标站点 robots、访问频率和服务条款

### 10.7 日历热点专栏规则

日历热点专栏是日历视图的辅助信息区，不是日历事件。

展示规则：

- 默认展示 3-5 条热点
- 只展示当天热点
- 标题最多两行，摘要最多两行
- 必须展示来源和刷新时间
- 点击热点进入详情或外部链接
- 日历离线或抓取失败时展示缓存状态

隔离规则：

- 热点不能进入冲突检测
- 热点不能触发日程提醒
- 热点不能被“删除日程”语音指令删除
- 热点专栏关闭只影响界面展示，不影响语音获取热点

### 10.8 每日简报规则

每日简报默认顺序：

1. 今日关键日程
2. 即将到来的提醒
3. 当日重要资讯

如果用户说“先说新闻”或“只听日程”，系统应按用户指令调整模块。

每日简报不能自动创建、删除或修改日程，只能读取和播报信息。

## 11. 安全、隐私与权限

### 11.1 权限原则

- 日历读取和写入权限必须明确授权
- 麦克风权限必须由用户主动授权
- 第三方日历 OAuth token 必须加密存储
- 用户可以随时断开第三方日历
- 用户可以删除语音记录和指令日志

### 11.2 语音数据策略

MVP 推荐默认不保存原始音频，只保存转写文本和解析结果。

如果为了调试保存音频，必须满足：

- 用户显式开启
- 支持按时间自动清理
- 支持手动删除
- 存储加密
- 日志脱敏

### 11.3 敏感信息处理

日历可能包含医疗、财务、工作机密等敏感信息。

要求：

- 日志中不要记录完整会议链接 token
- 对参与人、地点、标题进行可配置脱敏
- 调试日志按环境分级
- 生产环境默认关闭详细 NLU 中间日志

### 11.4 资讯请求隐私

资讯请求不应把用户完整日程发送给第三方资讯源。

要求：

- 只使用类别、地区、语言等低敏参数请求资讯
- 个性化推荐在本地或后端可信服务中完成
- 不把用户日历标题、参与人、地点透传给新闻 API
- 资讯点击跳转需要明确展示外部链接来源

## 12. 可用性与无障碍

### 12.1 免手持体验

- 支持语音播报完整闭环
- 用户不看屏幕也能完成创建、查看、删除
- 多候选选择可以用“第一个”“第二个”完成
- 长列表必须摘要播报

### 12.2 屏幕辅助

语音不是唯一通道。客户端应展示：

- 识别文本
- 系统理解的结构化字段
- 可点击确认/取消
- 候选事件列表
- 错误修正入口

### 12.3 错误恢复

用户可以说：

- “不是明天，是后天”
- “时间改成四点半”
- “标题改成复盘会”
- “算了”
- “重新来”

系统不能要求用户从头重复完整指令。

## 13. 质量指标

### 13.1 产品指标

- 单次语音创建成功率：>= 85%
- 添加提醒平均轮次：<= 1.3
- 添加事件平均轮次：<= 1.8
- 删除误操作率：<= 0.5%
- 查看日程响应时间：<= 1.5 秒
- 获取当日资讯响应时间：<= 2.5 秒
- 日历热点专栏首屏加载时间：<= 1.5 秒
- 热点手动刷新完成时间：<= 5 秒
- 每日简报首句响应时间：<= 3 秒
- 语音端到端响应时间：<= 3 秒

### 13.2 NLU 指标

- 意图识别准确率：>= 92%
- 日期时间解析准确率：>= 90%
- 删除候选召回率：>= 95%
- 资讯类别识别准确率：>= 90%
- 高风险操作确认覆盖率：100%
- 低置信度澄清触发率：100%

### 13.3 工程指标

- API 单元测试覆盖核心业务规则
- NLU 回归测试覆盖不少于 200 条中文语音指令文本
- 删除、修改、重复事件必须有集成测试
- 第三方日历同步失败必须有重试和降级测试
- 资讯来源失败时必须有缓存、降级或空结果提示测试
- 日历热点专栏必须有缓存命中、缓存过期、手动刷新和抓取失败测试

## 15. MVP 实施计划

### 15.1 第一阶段：文本核心链路

目标：

- 不依赖真实语音，先通过文本命令打通业务闭环

交付：

- 事件数据模型
- 文本指令 API
- 规则版意图识别
- 日期时间解析
- 创建、查看、删除、修改
- 当日资讯文本查询
- 热点实时抓取与短期缓存
- 操作确认和撤销
- 基础测试用例

### 15.2 第二阶段：语音闭环

目标：

- 接入 ASR 和 TTS，让用户可以通过语音完成核心操作

交付：

- 语音上传 API
- ASRService
- TTS 播报
- 客户端语音采集
- 会话状态管理
- 语音错误恢复
- 语音播报当日资讯和每日简报
- 保留语音播报热点能力，不因日历专栏上线而取消

### 15.3 第三阶段：同步与增强

目标：

- 让工具能接入真实日历生态，并具备稳定的资讯来源

交付：

- 系统日历或第三方日历适配器
- 双向同步策略
- 同步失败重试
- 冲突检测优化
- 重复事件完整处理
- RSS 或新闻 API 来源适配器
- 资讯缓存与失败降级
- 日历今日热点专栏
- 热点手动刷新接口

### 15.4 第四阶段：智能化增强

目标：

- 提升个性化和复杂场景处理能力

交付：

- 用户默认时间偏好
- 智能找空
- LLM 辅助 NLU
- 指令数据回归评测集
- 多语言或方言支持
- 个性化资讯频道和简报偏好

## 16. 验收标准

MVP 通过验收必须满足：

- 用户可以通过语音创建单次提醒
- 用户可以通过语音创建单次事件
- 用户可以通过语音查看今天、明天、本周日程
- 用户可以通过语音删除事件，且删除前必须确认
- 用户可以通过语音修改事件时间
- 系统能处理缺失日期或时间的澄清
- 系统能处理多个删除候选的选择
- 系统能检测创建或修改后的时间冲突
- 系统支持撤销最近一次写操作
- 所有日历写操作有操作日志
- 用户可以通过语音获取当日资讯
- 用户可以通过语音获取当日热点，且该能力不能被日历热点专栏替代或取消
- 日历当日视图必须展示“今日热点”专栏
- 今日热点必须支持实时抓取或手动刷新，并有短期缓存降级
- 用户可以通过语音获取日程与资讯组合的每日简报
- 资讯结果必须包含来源名称、发布时间和原文链接
- 至少 200 条中文指令回归测试通过
- 端到端语音响应在普通网络下不超过 3 秒

## 17. 风险与应对

### 17.1 时间解析错误

风险：

- “下周一”“这周日”等表达受地域和用户习惯影响

应对：

- 使用用户 locale 和周起始日配置
- 执行前复述完整日期
- 对低置信度时间表达进行确认

### 17.2 误删事件

风险：

- 语音识别或匹配错误导致删除重要日程

应对：

- 删除必须确认
- 多候选必须选择
- 支持撤销
- 使用软删除

### 17.3 语音识别错误

风险：

- 人名、地点、项目名容易识别错误

应对：

- 展示识别文本
- 支持口头修正
- 常用联系人和地点可加入个性化词表

### 17.4 第三方同步失败

风险：

- 本地成功但远端失败，导致数据不一致

应对：

- 本地保存同步状态
- 失败重试
- 明确提示用户
- 提供手动重新同步

### 17.5 隐私风险

风险：

- 日历和语音包含大量敏感信息

应对：

- 默认不保存音频
- token 加密
- 日志脱敏
- 支持用户删除历史指令

### 17.6 资讯来源不稳定或内容质量不足

风险：

- 新闻 API 限流、RSS 失败或来源内容重复
- 资讯摘要可能遗漏关键信息
- 用户把资讯摘要误认为完整事实核查

应对：

- 多来源适配和短期缓存
- 保留来源名称、发布时间和原文链接
- 低置信度摘要不做强结论表达
- 失败时明确提示“暂时无法获取资讯”，不编造内容
